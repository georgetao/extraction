Technical Discussion

Our Goals

To enable our project vision, we, the ExtrAction team, needed to build a system that is able to detect action items present in a given email, and then summarize them appropriately. We approached these two actions using two steps using two core NLP tasks: text classification and text summarization. Text classification is used to determine which sentences contain action items and then text summarization is used to make a succinct summary of each action item.

The Data

Our task requires us to create both a text classifier and a text summarizer, and therefore requires two distinct datasets to train each of these models. To curate these two specialized datasets, we draw natural language data from one primary source: the Enron Email Corpus1, a dataset of approximately 500,000 naturally written emails from the Enron Corporation in the early 2000s. It is currently the largest publicly available collection of naturally written language in an organizational setting.

Action Sentence Detection Dataset:

The text classification required for ExtrAction is classifying whether or not a particular piece of text contains an action item. The piece of text serving as our unit of analysis is the sentence. It is possible for multiple distinct action items to be expressed in a single sentence and also for a single sentence’s action item to depend heavily on previous sentences. However, according to our preliminary analyses and exploration, the sentence remains the unit most likely to contain exactly one distinct action item. Therefore, we proceeded to assemble a dataset of sentences and associated 1/0 labels indicating whether there is an action item present in the sentence. For short, we refer to a sentence containing an action item as an action sentence.


To create this dataset, we merged disparate labeled portions of the Enron Corpus. The most important of which is the EPA dataset2 for addressee tagging created by researchers at Microsoft. The dataset is a list of enron email messages containing highlighted sections which express a task which the sender asked the user to complete. We created a parser using Python to retrieve the sentence to which the highlighted task belongs as well up to 3 sentences that came before the action sentence. To balance out the positives, we pulled negative examples (non-action sentences) from Email Intent Dataset3 and from Parakweet Labs the UC Berkeley Enron Email Analysis Project.



Action Item Summarization Dataset:

The next dataset we created associated an action sentence and its up to 3 sentences that preceded it with a summary of the action item. We include up to 3 preceding sentences because they tend to contain details that are important to fully understanding the action item.


To curate this dataset, members of our team together annotated approximately 4,000 action sentences with action item summaries. To minimize influence of members’ personal linguistic signatures in the summaries, we collectively adhered to a set of agreed upon guidelines when summarizing the action items. The primary principles to which the members agreed emphasized clarity, conciseness, and attractiveness (using words present in the text).


Finally, to supplement our dataset of 4,000 instances we used the Wikihow Summarization Dataset4, consisting of article paragraphs and their 2nd person, prescriptive summaries. We use nearly 1 million instances from the WikiHow dataset for pre-training before fine-tuning the model on our own curated dataset.

Classification Model

We chose to use fastText, a hierarchical text classification model developed in the Facebook AI Research Lab, after comparing it to multiple baseline and deep learning models. We found fastText to be on par with the deep learning models in terms of accuracy while also being extremely fast and easy to train.


For ExtrAction, computational efficiency is especially important because the model will have to evaluate in real time on multiple sentences. fastText is efficient on large datasets because it is a hierarchical classifier. Categories are arranged on a tree which reduces training and testing time from linear to logarithmic. Additionally, the tree is built using the Huffman algorithm which leads to more computational efficiency by allowing the model to exploit natural class imbalances.

Summarization Model

After considering both performance and pre-existing libraries available to us, we eventually settled on using a Pointer-Generator Network, invented by Abigail See in 2017, to use as a model for summarizing action items. This choice was our first because some of the Pointer Generator’s signature features are particularly favorable to ExtrAction’s business context.


The Pointer-Generator is both an extractive and an abstractive summarization architecture. At each step in producing a summary, the model chooses between using a token from the input text or one from the vocabulary it is trained with. Experiments in the literature have shown that, more often than not, the pointer generator chooses to “point” or copy a token from the input sequence. This feature is critical to our task because the model is likely to see important tokens which are outside its known vocabulary quite often in the context of business. For example, if an action sentence says “Please send a copy of our agreement to Rowan Cassius”, chances are high that the names Rowan and Cassius are out of vocabulary (OOV). This is not a problem for the pointer generator because it can simply point to “Rowan” and then “Cassius” when producing a summary and does not need to have decent vector representations for every name.


Additionally, we made customizations to the vanilla pointer-generator described by Abigail See by using pre-trained GloVe word vectors, action and context segment embeddings and customized named entity merging and coding.

User Interface

When it comes to Extraction’s user interface, we wanted to stay true to Extraction’s primary mission of simplifying the lives of its users. As a result, our UI design prioritizes a simple and straightforward user experience and all the features we’ve decided to add were deemed necessary and irreplaceable.


The primary feature of Extraction is the generated list of action items. We wanted the main view of the extension to focus on the list of action items to make it as easy as possible for users to access this information. We wanted to let users manipulate the action items however they chose to do. Every feature we decided to include supplements this list, while not simultaneously disrupting the simplicity of the extension.




We originally planned on creating a system that could pull a batch of emails at a time and automatically turn them into action items. However, gmail authentication and scraping proved to be a difficult and imprecise task, so we decided to include the text box to allow users to correct possible errors or simply paste emails into the text box and forego the scraping process if they so chose to.


Our final user interface maintains the original design concept of simplicity, while also including all the features we deemed necessary to create a functional and exportable to-do list.
